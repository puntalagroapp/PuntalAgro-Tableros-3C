¿Qué es esto?

PuntalAgro Tableros 3C es un sistema web de gestión agropecuaria multi-empresa. Permite a productores agropecuarios gestionar campos, lotes, campañas, insumos, órdenes de trabajo, planes de uso del suelo y tableros de decisión.

El nombre "3C" hace referencia a la migración de una arquitectura standalone (todo en el browser) a una arquitectura de 3 capas: Frontend → API → Base de datos. La migración es incremental: algunos tableros ya leen de la API; otros todavía usan datos hardcodeados en JS.

El frontend está escrito en ES5 puro (var, function, XHR callbacks) — sin bundler, sin TypeScript, sin framework. El cliente del frontend puede hacer cambios. La lógica sensible vive en el backend.

Base de datos

El archivo database/init.sql se ejecuta automáticamente al crear el volumen Docker por primera vez. Incluye el esquema completo + datos seed (usuarios demo, empresas de prueba, campañas, etc.).

Para aplicar migraciones sobre una base existente:

docker exec -i pa_postgres_db psql -U postgres -d puntalagro_tableros_3C \
  < database/migrations/001_add_missing_fks.sql

Autenticación y sesiones

Flujo de login

// login.html → POST /api/auth/login
Headers: { Authorization: "Bearer <API_KEY>", Content-Type: "application/json" }
Body:    { email: "...", password: "..." }

// Respuesta exitosa
{ sesion: { token, usuarioId, nombre, email, rol, clienteId, expiraEn } }

// El frontend guarda en localStorage
localStorage.setItem('pa_sesion_activa', JSON.stringify(sesion))

Autenticación de requests

Todos los endpoints protegidos requieren:

Authorization: Bearer <token>

El backend verifica que el token exista en la tabla sesiones y que expira_en > NOW().

Estructura de un permiso

-- Un registro en tabla permisos:
{
  usuario_id:   "u_maria",
  empresa_id:   "e_1",
  campo_ids:    ["c_1"],   -- [] significa acceso a todos los campos
  herramientas: ["tablero_uso_suelo", "tablero_insumos_ot"],
  nivel:        "ver"      -- ver | cargar | administrar
}

Cuando campo_ids es un array vacío {}, el usuario tiene acceso a todos los campos de la empresa. Si contiene IDs específicos, los lotes se filtran por esos campos al cargar cualquier tablero.

Patrones y convenciones

IDs como TEXT

Todos los IDs son TEXT generados en el frontend con window.crypto.getRandomValues() (función uid(prefix) en pa-core.js). No se usan sequences excepto en permisos.id y tableros.id. Esto facilita la inserción offline y la migración desde el sistema standalone.

PK compuesta en S4 y S5

Las tablas de maestros y datos operativos usan PRIMARY KEY (id, empresa_id). Esto permite que el mismo ID lógico exista en múltiples empresas sin conflicto. Las FK hacia estas tablas son necesariamente compuestas:

FOREIGN KEY (insumo_id, empresa_id) REFERENCES insumos(id, empresa_id)

Con MATCH SIMPLE (default), si alguna columna del lado referenciante es NULL, la FK no se evalúa. Útil para columnas opcionales como tercero_id en OTs.

JSONB para maestros

Los maestros de S4 almacenan sus campos en una columna datos JSONB. El id y el empresa_id también se guardan dentro del blob para simplificar la serialización desde JS. Los índices UNIQUE usan expresiones sobre el JSON:

CREATE UNIQUE INDEX ON terceros (empresa_id, lower(datos->>'nombre'));

Tableros blob (S6) vs. tablas normalizadas

Los tableros que aún no migraron guardan su estado completo como un blob JSONB en la tabla tableros, con una clave textual como plan_uso_suelo_e_1. No es posible filtrar por campo dentro del blob — por eso los tableros que necesitan filtrado por permisos de campo deben leer lotes desde el endpoint normalizado y no confiar en el blob para eso.

tablero_uso_suelo ya migró a este modelo: carga lotes desde la API (filtrados), y solo guarda en el blob el plan (ambiente, actividades, tipo). Los tableros pendientes de migrar: tablero_insumos_ot, ProgramaSiembra, Fitosanitarios, tablero_agro, tablero_evolucion, tablero_hacienda, tablero_labores.

Filtrado por permisos de campo

-- Ejemplo: lotes filtrados por campo_ids del permiso del usuario
SELECT l.id, l.nombre, l.ha, l.campo_id
FROM lotes l
WHERE l.empresa_id = $empresaId
  AND ($campoIds = '{}' OR l.campo_id = ANY($campoIds::text[]))

campo_ids = '{}' significa acceso a todos los campos. Si tiene IDs específicos, se filtra.

Seguridad

MedidaImplementación
CORSVariable ALLOWED_ORIGINS (coma-separada). Si está vacía, acepta todo (solo dev). Configurar en producción.
Rate limiting loginEn memoria: 10 intentos / 15 min / IP. Se reinicia al reiniciar el proceso. Para producción considerar Redis.
API key en loginBearer token compartido entre frontend y backend (API_KEY). Hardcodeado en login.html; actúa como primera barrera contra bots.
Tokens de sesióncrypto.randomBytes(32) — 256 bits de entropía. TTL: 24 horas.
Password hashingcrypto.scrypt(password, salt_16bytes, 64). Formato: salt_hex:key_hex.
Errores internosTodos los catch devuelven "Error interno del servidor" — sin stack traces al cliente.
Validación de inputEmail máx 254 chars, password máx 200 chars, tipo string obligatorio. empresaId requerido en todos los endpoints de tablero.
Generación de IDs frontendwindow.crypto.getRandomValues() en lugar de Math.random().
